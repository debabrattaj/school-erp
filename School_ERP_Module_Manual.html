Authentication & Access Control

Login

Purpose: This is the entry screen for the whole application. Every person who uses the ERP — administrators, principals, accounts staff, teachers, and (through a separate "portal" landing area) parents and students — signs in here with a school account code, email address, and password. Depending on the role, and depending on whether the account has Two-Factor Authentication (2FA) turned on, the screen may also ask for a 6-digit authenticator code.

Tips/Notes

  • Where a user lands after signing in depends on their role: Parents and Students go to a separate Family/Student "portal" area; built-in staff roles (Admin, Principal, Accounts, Teacher) go to the main dashboard; a custom role lands on the first module they have any permission for, or the dashboard if none is defined.
  • Any status message (errors or the 2FA prompt) automatically disappears from the screen after 2 seconds, so if a message appears and vanishes quickly before it's read, re-submit the form to see it again.
  • 2FA is optional per user and is turned on/off from the Profile & Security screen (see below), not from the login screen itself.

Master Data & Academic Structure

This section covers the screens that set up the building blocks used across the rest of the School ERP: the master dropdown lists, the school profile/settings, academic years, classes and sections, subjects, and the mapping between classes and exams. These are typically configured by the school Admin/Principal at the start of each academic year, and changes here ripple into Students, Teachers, Attendance, Fees, Exams, Marks, Report Cards, and more.

Master Data

Purpose

Master Data is the central control panel for all the dropdown option lists used throughout the ERP — things like the list of Classes, Sections, Departments, Genders, Blood Groups, Fee Types, Exam Types, and more. School Admin staff use this screen once during setup (and occasionally afterward) to add, edit, deactivate, or remove the choices that appear in dropdowns on other screens (Students, Teachers, Classes, Fees, Attendance, Exams, etc.). Only users with the Admin role can add, edit, or delete entries here; Principal, Accounts, and Teacher roles can view the lists (e.g., where they are used as read-only dropdown sources elsewhere) but the Master Data screen itself is generally an Admin setup tool.

Why this page matters for the rest of the system: many other modules do not let staff type free text for fields like "Class," "Section," "Department," or "Academic Year" — instead they show a dropdown that is populated directly from the values you create here. For example:

  • The Class dropdown used on the Classes screen (and elsewhere) is fed by the "Class" category here.
  • The Section dropdown on Classes is fed by the "Section" category here (plus any sections already in use).
  • The Academic Year dropdown on the Academic Years screen is fed by the "AcademicYear" category here.
  • Categories such as Department, Gender, Blood Group, Fee Type, Attendance Status, Exam Type, Employment Type, Nationality, Transport Route, Library Category, Inventory Category/Unit, Salary Grade, Student Status, and House work the same way for their respective modules (Students, Teachers, Fees, Attendance, Exams, Transport, Library, Inventory, Payroll, etc.).

If a dropdown elsewhere in the ERP looks empty or is missing an option you need, the fix is almost always: come to Master Data, pick the right category on the left, and add the missing value.

Tab: Schools

Purpose

Lets the platform owner see every school registered on the platform, create brand-new school accounts, control which modules (features) each school can use, suspend or reactivate a school's access, and reset a school's admin login credentials (a support tool for when a school is locked out).

Account Code Text Yes User input (free text) A unique short code used internally to identify the school's own separate database/tenant. Must be alphanumeric (dashes and underscores are also allowed)
Admin Name Text Yes User input (free text) Name of the first admin user created inside the new school
Admin Email Text (email format) Yes User input (free text) Login email for the new school's first admin account

Internally the system also records "Country" (defaults to "India") and "Timezone" (defaults to "Asia/Calcutta") using fixed default values that are sent automatically with the form — there is no visible input for these two on the Create School panel.

Tips/Notes

  • Suspending a school is a serious, immediate action — the app explicitly warns "All logins will stop" and requires the operator to confirm via a popup before proceeding. Every user at that school (admins, teachers, front-desk staff, parents/students if portal is enabled) will be locked out until the school is reactivated.
  • Creating a school is essentially irreversible through this screen — there is no "delete school" button anywhere in the console. Once created, a school's own database is provisioned immediately, so account codes cannot be reused for a different school later.
  • The "Modules" checkboxes let the platform owner hide entire sections of the ERP from a school (e.g. disable Hostel or Library for a day school that doesn't need them) without deleting any of that school's existing data — turning a module back on later restores access to whatever was already there.
  • The "Reset School Admin Login" tool doubles as a way to add a brand-new admin login to a school (if the email typed in doesn't match an existing login) — useful if a school's only admin account was somehow lost.
  • Success/error banners auto-dismiss after 2 seconds, so read them promptly.
  • If the platform owner's session has expired or is invalid, any data-loading action in this console will automatically sign the operator out and return to the Owner Login page.

Errors & validation table

Error message (exact or paraphrased) When it happens How to fix it

Tips/Notes

  • Assigning a new subscription automatically supersedes any previous one — the old subscription is kept in the history for record-keeping but flips to "Replaced" status and no longer counts toward revenue or expiry warnings.
  • Leaving "Amount Paid" at 0 is a shortcut to auto-charge the plan's standard list price for the chosen billing cycle — useful for standard renewals, but be careful if a school is meant to get a discounted or negotiated rate: type the actual amount in that case.
  • The "Duration (months)" field is always interpreted as 30-day months (not actual calendar months), so a 12-month subscription expires 360 days after the start date, not exactly one calendar year later.
  • Rows in both the summary cards and the tables are color-coded — amber for "expiring soon" (30 days or less remaining) and red for "expired" or overdue — making it easy to scan for schools needing attention.

Tips/Notes

  • The audit log is capped at showing the 200 most recent matching entries at a time — for older history, narrow the filters (e.g. by school or by actor email) to find specific events, since there is no further pagination control on this screen.

  • Entries with a status code of 400 or higher (shown in red) represent failed or rejected requests — useful for spotting repeated failed attempts (e.g. a user trying and failing to access something they shouldn't).
  • This log spans all schools on the platform in one place — only the platform owner can see this cross-school view; individual schools cannot see other schools' audit entries.

This is an operational/technical safety-net feature, but it is the platform owner (not a developer) who is expected to use it from this screen when needed.

General notes about the Owner Console (applies to all tabs)

  • Access control: Every action in this console requires a valid platform-owner sign-in token. If that token is missing, invalid, or has expired while using the console, the app automatically signs the operator out and returns to the Owner Login page the next time it tries to load data.
  • Confirmation dialogs: Suspending a school is the only action in this console protected by an explicit "Are you sure?" confirmation popup, reflecting how disruptive it is (it immediately blocks all logins at that school). Activating a school, creating a school, assigning a subscription, sending a notification, resetting a password, and triggering a backup do not show a confirmation prompt — they take effect as soon as the button is clicked, so review the form carefully before submitting.
  • No delete functionality: There is no button anywhere in this console to permanently delete a school, a plan, a subscription record, or a notification. The only way to cut off a school's access is to Suspend it; historical records (old subscriptions, sent notifications, audit entries) are kept indefinitely for the record.
  • Success/error banners: Every action shows a temporary banner message at the top of the page confirming what happened (or what went wrong), which automatically disappears after 2 seconds.
